# Chat HTTP Body Timeouts

## Conclusion

The current April 21 rerun for `BurntSushi__ripgrep-2209` did repeatedly hit `chat_http_request_error` with `phase="body"` after already receiving HTTP 200 headers. The immediate cause is the split request path in `ploke-llm`: the request succeeds through `send()` and header receipt, then later fails in `resp.text().await` while reading or decoding the body stream. In this run, those failures were recovered by the inner HTTP retry loop, so the observable effect was delay, not loss of the completed rerun.

The code does still have an outer retry/abort path if the inner HTTP attempts are exhausted. That means body timeouts can strand a run in principle, but the current completed rerun artifacts only prove added latency.

## Cause -> Effect Narrative

1. The eval harness starts the benchmark with a chat policy that allows outer request retries and backoff if `chat_step` returns a retryable error.
2. Inside `ploke-llm`, the HTTP request reaches the provider and receives headers successfully, which is why the log shows `status=200` before the failure.
3. The next step is still to read and decode the body. That happens later in `resp.text().await`, so the request can succeed at the HTTP/status layer and still fail at the body-read layer.
4. In the current log, each failure happens at about `elapsed_ms=45003` with `timeout_secs=45` on the request start line. The most likely reading is that the request deadline expired while waiting for the body stream to finish, and `reqwest` surfaced that as `error decoding response body`.
5. Those failures are handled first by the inner `ploke-llm` retry loop, which emits `chat_http_request_error` and `chat_http_retry_scheduled` and then reissues the same provider call.
6. Only if all inner attempts fail does the error escape into `ploke-tui`, where it becomes `HTTP_BODY_TIMEOUT` and may trigger the outer `chat_step failed; retrying` loop. If that outer retry budget is exhausted too, the session can abort and the eval run can be left incomplete.
