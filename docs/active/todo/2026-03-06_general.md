# todo March 6, 2026

## Clean up User-Facing Issues

The application is mostly usable on individual crates now. Next, we want to
focus on the user experience by addressing potential sticking points or poor
experiences the user might have when engaging with the application.

### Conversation UI Filtering

- see further details in `../plans/conversation-ui-filtering.md`

We want to check our previous work on filtering the conversation view in the
UI. By default right now there is a lot of data presented to the user that
clutters up the UI. While this information is helpful in debugging, and we want
to make it possible to expose feedback from the different systems producing
these messages, we also want to keep the UI tight and focused on the most
relevant information for the user.

One example is in the feedback from the tools called by the LLM. Right now we
might see the number of items returned in the search result and similar info,
but instead we should present the user with minimal tool info, instead just
letting the user know the tool is being called, whether through the message
returned by the LLM during the tool call, or through a "calling tool <tool
name>" or similar.

### UI and command discoverability

There are two main points to cover here:

1. Command Completion

Command completion is currently mostly impolemented but might be missing some
commands, and some of the commands have placeholder descriptions in the
completion popup. We should ensure all commands are present in the completion
feature, and that they all have accurate and brief descriptions.

2. Command Discoverability

We currently have the "/help" command (also accessibly in Normal mode with the
hotkey `h`), but this current presents the user with a wall of text that might
be difficult to take in or navigate. We want to give the user an efficient way
of finding the functionality they are looking for.

To achieve this, we want to implement an overlay that has:
  - A search bar
  - A list of commands and descriptions of functionality for the command
    - The list should be scrollable
  - The list should be focused on an element, and with the up/down arrow keys
  and the `Ctrl+p` and `Ctrl+p` keys navigable. When the user presses `Enter`
  on a focused command, it should autocomplete into the input box.
  - The search bar should live-update the elements in the list.
  - The search should include not only the command, but also the description of
  the command.

Once this has been implemented, if feasible we should use either the bm25
search already present in `ploke-db` on the command list to make the search bar
use fuzzy search of the commands and descriptions. There may be an example of
how to do this already in the "context search" overlay.

### Conversation UI presentation

Two main points to cover:

1. Conversation Message Spacing

Currently there is no or minimal spacing between messages in the conversation UI. This leads to too much text being presented to the user without visual ques to break up the messages in a helpful way.

Instead, we should add a configurable "spacing" between messages in the conversation, and set the default to 1 row.

- Stretch goal: message grouping

Having spacing between messages is good, but even better would be grouping
message items. For example, if there are multiple tool calls followed by a
final return message from the LLM, perhaps we leave no spacing between the tool
calls, but then add a space for the final LLM message. This creates meaningful
boundaries that guide user attention to logically grouped elements of the
conversation.

2. Color Scheme and message styling

Currently there are relatively loud differences in the colors used for the
diferent sources of messages such as System vs Assistant vs User vs SysInfo
messages. This is mainly an artifact of our implementation process and has been
helpful in debugging.

Now that we are cleaning up the product to be more user-friendly, we want to
change the colorscheme to be more subtle, using the same color for all
messages, and using a light gray background for User messages, and dimmed color
where currently used. We will not address the markdown and code highlighting in
this PR.

However, we do not want to completely remove the previous coloring system, and
preserve it in a toggle-able debug mode that presents the same stark visual
differences in color that we are currently using.

- Stretch goal: color config

Most of our color choices are currently hard-coded, but it would be good both
for user customization and for internal developer iteration to have a way to
select the color scheme used for the application.
  - A color config that loads from a `.toml` file.
