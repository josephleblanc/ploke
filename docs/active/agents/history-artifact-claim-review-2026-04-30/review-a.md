Verdict: The commit moves in the right direction by deriving a backend clean-tree key after installing/building the selected successor artifact and admitting that key through `Crown<Ruling>` into the sealed block claims. That is real sealed evidence, and the new claim envelope is committed into the block hash. It does not yet encode the core handoff admission invariant: the successor can still enter the parent path through invocation/scheduler checks without verifying the sealed History head or matching its current clean artifact tree against the admitted artifact claim. The implementation is therefore still a recording step, not a ruling-path gate.

Findings:

1. High: Artifact claim admission is not used for successor admission, so the local Crown/History eligibility claim remains unencoded. The docs state the target gate clearly: startup must establish `ProducedBy(SelfRuntime, CurrentArtifact)` and `AdmittedBy(CurrentArtifact, Lineage, Policy, History)` before entering the ruling path, and the live path does not yet enforce it ([history.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/history.rs:60), [history.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/history.rs:125), [history.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/history.rs:273)). The new live seal path admits the artifact claim and appends the sealed block ([prototype1_process.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_process.rs:903), [prototype1_process.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_process.rs:927)), but successor acknowledgement still validates only the invocation, parent identity, active root path, and scheduler continuation decision ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3056), [prototype1_process.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_process.rs:384)). There is no read of the current sealed head, extraction of `claims.artifact`, derivation of the successor checkout `TreeKeyHash`, or comparison before `parent.ready()` ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3116), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3128)). This means the commit supports later admission, but does not itself make "only a runtime whose current Artifact matches the successor Artifact committed by the sealed History head may enter the ruling parent path" true.

2. Medium: The new `ArtifactLocator` verification boundary is tautological unless a caller separately supplies the current checkout tree key. `ArtifactLocator::locate` accepts any stored `TreeKeyHash` and returns an `Artifact`, and `digest` hashes that same stored key ([history.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/history.rs:987)). `Verifiable::verify_with` then recomputes the digest from `self.key`, not from an observed checkout ([history.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/history.rs:1047)). That proves the flattened claim was internally well-formed, but not that the runtime's current clean tree matches the claim. This is acceptable as a storage envelope, but the name and API make it easy for a future caller to treat `artifact::<ArtifactLocator>().verify_with(&ArtifactLocator)` as artifact admission when the missing step is "derive current `TreeKeyHash` through `WorkspaceBackend::clean_tree_key` and compare it to the sealed claim key."

3. Medium: Artifact claims are not required by the seal transition type. `SealBlock::from_handoff` still creates a seal with `Claims::empty_unchecked()` ([history.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/history.rs:1847)), and `LockCrown::seal_block` remains a crate-visible path that seals without admitting any claims ([inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:167), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:195)). The changed live call uses `seal_block_with` correctly, but the type boundary still permits a handoff block with no artifact claim. For an invariant this central, the successor artifact claim should be part of the handoff transition input/output shape, not an optional callback mutation.

Structural Naming Notes:

- `seal_block_with` is mechanically named and exposes a generic callback over `&Crown<Ruling>`. It would be harder to misuse as a structural transition such as `seal_handoff(open, SuccessorArtifactClaim, seal_fields)` or a `HandoffSeal` carrier that already contains the required admitted artifact claim.
- `ArtifactLocator` sounds like it locates a real artifact, but currently it only replays a digest over a stored `TreeKeyHash`. Consider naming around the actual structure, for example `TreeKeyArtifactClaim`, or make the locator carry a backend/current-checkout observation so the name is true.
- Legacy flattened names remain documented as legacy evidence rather than History kinds, which is good: `ActiveCheckoutAdvancedEntry` and `SuccessorHandoffEntry` are explicitly called out as labels to normalize into structural facts ([journal.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/journal.rs:235), [journal.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/journal.rs:252)).

In the same patch or the next one, replace optional claim mutation during sealing with a required handoff seal carrier so live handoff cannot produce a sealed successor block without an artifact claim.
