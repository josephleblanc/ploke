# Single-Ruler Readiness B

Date: 2026-05-01

Scope: Prototype 1 long-horizon single-lineage loop reliability, with emphasis on practical 1000-generation single-ruler execution. Source files were inspected only; no source edits were made.

## 1. Verdict For Running More Loop Experiments Now

Run more short loop experiments now, but do not treat the current implementation as ready for an unattended 1000-generation run.

The implementation has moved in the right direction for a local single-ruler canary: successor handoff installs the selected Artifact into the stable active checkout before successor launch, seals/appends a minimal History block before spawning the successor, and successor startup checks the current clean tree and surface against the sealed head (`crates/ploke-eval/src/cli/prototype1_process.rs:610`, `crates/ploke-eval/src/cli/prototype1_process.rs:966`, `crates/ploke-eval/src/cli/prototype1_process.rs:990`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:504`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:512`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:518`).

The same code is still explicitly local and partial. The docs say not to assume one global Parent, not to couple worktree layout to semantic Tree identity, and not to make successor authority global authority (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:18`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:24`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:25`). Tree and lineage are distinct, and the same Artifact/tree key may be referenced by more than one lineage head under policy; authority conflicts are about advancing the same History lineage head without Crown/lock/consensus/fork-choice, not shared Artifact identity (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:145`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:147`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:150`). This correction is reflected in `history.rs`: lineage authority is keyed by History lineage coordinate, not branch/worktree/process/runtime/artifact identity alone (`crates/ploke-eval/src/cli/prototype1_state/history.rs:193`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:195`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:197`).

## 2. Top Concrete Operational Risks For 1000 Generations

1. Non-atomic mutable JSON projections are still load-bearing operationally. Scheduler, node records, runner requests/results, branch registry, invocations, successor-ready, and successor-completion records are written with direct `fs::write` calls (`crates/ploke-eval/src/intervention/scheduler.rs:372`, `crates/ploke-eval/src/intervention/scheduler.rs:405`, `crates/ploke-eval/src/intervention/scheduler.rs:424`, `crates/ploke-eval/src/intervention/scheduler.rs:438`, `crates/ploke-eval/src/intervention/branch_registry.rs:202`, `crates/ploke-eval/src/cli/prototype1_state/invocation.rs:344`, `crates/ploke-eval/src/cli/prototype1_state/invocation.rs:475`, `crates/ploke-eval/src/cli/prototype1_state/invocation.rs:507`). A crash, kill, or full disk event can leave parse failures or old/new split state. The append-only transition journal is better because it appends and syncs each line (`crates/ploke-eval/src/cli/prototype1_state/journal.rs:606`, `crates/ploke-eval/src/cli/prototype1_state/journal.rs:616`, `crates/ploke-eval/src/cli/prototype1_state/journal.rs:621`), but the live controller still depends heavily on overwrite projections.

2. History append is not transactional across segment, secondary indexes, and heads projection. `FsBlockStore::append` writes the sealed block, then by-hash index, then by-lineage-height index, then overwrites `heads.json` (`crates/ploke-eval/src/cli/prototype1_state/history.rs:861`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:862`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:863`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:872`). It detects some inconsistent projections later, such as missing heads with stored blocks and missing lineage heads (`crates/ploke-eval/src/cli/prototype1_state/history.rs:707`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:712`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:883`), but a long run needs recovery/rebuild, not just rejection.

3. There is no process/file lease for the local lineage head. The store does stale-state checks (`crates/ploke-eval/src/cli/prototype1_state/history.rs:839`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:840`) and sparse local lineage proofs (`crates/ploke-eval/src/cli/prototype1_state/history.rs:1152`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:1160`), but the docs correctly say this is not process uniqueness or global consensus (`crates/ploke-eval/src/cli/prototype1_state/history.rs:30`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:32`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:140`). Two local parents started accidentally before/around handoff remain an operational hazard.

4. Cleanup exists but is best-effort and not yet a complete recovery system. Child worktrees and build products are removed after compile failure, failed result, and success (`crates/ploke-eval/src/cli/prototype1_process.rs:1808`, `crates/ploke-eval/src/cli/prototype1_process.rs:1880`, `crates/ploke-eval/src/cli/prototype1_process.rs:1899`). Successor handoff also removes the temporary successor worktree before active-checkout install and deletes node-local child build products (`crates/ploke-eval/src/cli/prototype1_process.rs:542`, `crates/ploke-eval/src/cli/prototype1_process.rs:548`). However, a failure between commit, cleanup, checkout switch, History append, and successor spawn can strand worktrees, branches, bins, targets, or a retired parent with no live successor. The monitor inventory itself labels the prototype root as accumulating/mutating and node worktrees/build products as temporary (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:1745`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:1830`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:1835`).

5. Default/policy limits are not 1000-generation settings. `Prototype1SearchPolicy::default` is one generation and 32 nodes (`crates/ploke-eval/src/intervention/scheduler.rs:23`, `crates/ploke-eval/src/intervention/scheduler.rs:26`, `crates/ploke-eval/src/intervention/scheduler.rs:27`), and continuation stops when `next_generation > max_generations` or total nodes reach the cap (`crates/ploke-eval/src/intervention/scheduler.rs:646`, `crates/ploke-eval/src/intervention/scheduler.rs:648`). This is not a correctness bug, but a 1000-generation experiment needs explicit policy and disk budget.

6. Operator observability is broad but still projection-heavy. Reports expose continuation, staged nodes, selected branch, branch evaluations, and pending stages (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:4489`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:4590`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:4653`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:4669`). The journal report explicitly marks legacy journal material as a projection (`crates/ploke-eval/src/cli/prototype1_state/report.rs:303`, `crates/ploke-eval/src/cli/prototype1_state/report.rs:335`, `crates/ploke-eval/src/cli/prototype1_state/report.rs:343`). For 1000 generations, operators need a compact health view over head, active checkout, child/successor liveness, cleanup debt, disk growth, and last durable recovery point.

## 3. Top Invariant Risks For 1000 Generations

1. The current handoff test path still promotes any completed child evaluation as successor-eligible, even when evaluator disposition is `Reject`. The code labels this as a temporary short-circuit and says it was for live handoff testing (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3419`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3423`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3426`). That is the top invariant risk for meaningful long-horizon improvement: the loop may remain mechanically alive while policy quality drifts.

2. Startup admission is partial, especially bootstrap. The docs say the uniform startup shape is `ProducedBy(SelfRuntime, CurrentArtifact)` plus `AdmittedBy(CurrentArtifact, Lineage, Policy, History)` (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:85`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:88`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:91`). Implementation has a predecessor gate against sealed History, but gen0/setup does not append a genesis block until the first live handoff (`crates/ploke-eval/src/cli/prototype1_state/history.rs:366`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:367`). For single-ruler canaries this is manageable; for robust restart/admission it is a gap.

3. History blocks are minimal and evidence-poor. The current live block can be sealed through Crown and includes an admitted Artifact claim, but `SealBlock::from_handoff` starts with empty claims (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2476`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2486`), and current docs say live `Parent<Ruling>` is not yet the only writer of open block entries, uniform admission is missing, ingress import is missing, and signatures/consensus are absent (`crates/ploke-eval/src/cli/prototype1_state/history.rs:264`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:266`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:267`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:269`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:270`). This is not a blocker for mechanical canaries, but it is a blocker for strong claims about selection evidence over 1000 generations.

4. `Parent<Ruling>` and actor identity are still partly data-supplied at the History boundary. `Crown<Ruling>` gates block opening/admission/sealing, but the docs and code note that actor identity is still passed as data until `Parent<Ruling>` supplies it structurally (`crates/ploke-eval/src/cli/prototype1_state/history.rs:3023`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:3029`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:3094`). This weakens auditability more than immediate single-process execution.

5. Surface commitment is useful but intentionally narrow. The policy-bearing immutable surface is `crates/ploke-eval`, mutated surface is tool-description files, and ambient is empty (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:101`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:104`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:227`). The backend hashes tracked `crates/ploke-eval` files, all `ToolName::ALL` description paths, and an empty ambient set (`crates/ploke-eval/src/cli/prototype1_state/backend.rs:1217`, `crates/ploke-eval/src/cli/prototype1_state/backend.rs:1222`, `crates/ploke-eval/src/cli/prototype1_state/backend.rs:1245`, `crates/ploke-eval/src/cli/prototype1_state/backend.rs:1248`). This is correct for ordinary tool-text mutation, but it is not a general protocol-upgrade mechanism and should not be treated as a complete policy surface.

Selected-successor runtime identity should not be upgraded into global authority. The sealed block records `selected_successor`, and invocation/ready files verify transport coordinates, but the real local authority gate is current lineage head plus current clean Artifact tree/surface, not exclusive ownership of a runtime/node/artifact identity (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2108`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2950`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2983`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3069`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3118`).

## 4. What Should Be Patched Before Long-Horizon Runs

1. Remove or quarantine the temporary "any completed child can continue" shortcut before any run whose result is meant to say anything about improvement.

2. Add a local lineage lease/lock around `prototype1-state` and `FsBlockStore::append`, or otherwise make duplicate local parents fail before they can mutate the same configured History lineage head.

3. Replace direct projection overwrites with atomic write-rename plus directory fsync where possible, and add recovery/rebuild commands for scheduler, branch registry, node mirrors, runner results, and History indexes from the append-only or content-addressed source of truth.

4. Make History append transactional enough for local crashes: either single authoritative append file plus rebuildable indexes, or locked append with durable staging/commit markers and recovery.

5. Add a cleanup/reconciliation command that reports and optionally removes stale node worktrees, node-local `target/`, child binaries, orphaned git worktrees, stale branches, dead successor attempts, and inconsistent invocation/result pairs.

6. Add an operator readiness/health report for the exact long-run state: current lineage head, active checkout branch/tree, parent identity, latest sealed block, latest continuation decision, live or terminal successor status, cleanup debt, disk usage, and projection consistency.

## 5. What Can Wait While Focus Shifts Elsewhere

Distributed consensus, global finality, cryptographic signatures, multi-ruler fork choice, and cross-lineage canonicalization can wait for now. The current docs already narrow the claim to local single-ruler authority (`docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:360`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:379`).

Full artifact-local provenance manifests, stochastic evidence roots, risk-budget semantics, and complete ingress import can also wait if the next work is only mechanical loop hardening. They become blocking before claims about durable improvement, reproducible evaluation, or long-horizon selection quality.

General protocol-upgrade support for changing `crates/ploke-eval` can wait. Ordinary Prototype 1 succession should continue to keep that surface immutable until the upgrade transition is explicitly modeled (`docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:180`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:201`).
