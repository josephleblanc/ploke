# S2D Metrics Backfill Prototype Report

## implemented

- Added a tiny sample companion row in [2026-04-12_S2D_sample.rows.jsonl](./2026-04-12_S2D_sample.rows.jsonl) from the real run directory `BurntSushi__ripgrep-1294`.
- Added a regenerated markdown excerpt in [2026-04-12_S2D_sample.rendered.md](./2026-04-12_S2D_sample.rendered.md) to show the companion-to-ledger render step.
- Kept the prototype intentionally narrow: it proves the backfill/render shape on one real sample without building ingestion automation.

## claims

- [AC1] The proposed JSONL-companion flow can be seeded from a small real run sample using the current run-directory artifacts.
- [AC2] A markdown ledger excerpt can be regenerated from that sample row without introducing additional machinery.
- [AC3] The prototype makes the boundary explicit: canonical `experiment_id` / `manifest_id` and some validity fields remain hypothetical in the current corpus.
- [AC4] The next step is bounded: add a tiny generator only after target-converged formal manifests exist, or keep the surrogate-key adapter narrow if this corpus remains task-id keyed.

## evidence

- Sample input set:
  - `~/.ploke-eval/runs/BurntSushi__ripgrep-1294/run.json`
  - `~/.ploke-eval/runs/BurntSushi__ripgrep-1294/execution-log.json`
  - `~/.ploke-eval/runs/BurntSushi__ripgrep-1294/agent-turn-summary.json`
  - `~/.ploke-eval/runs/BurntSushi__ripgrep-1294/indexing-status.json`
  - `~/.ploke-eval/runs/BurntSushi__ripgrep-1294/snapshot-status.json`
- Prototype artifacts:
  - [sample JSONL row](./2026-04-12_S2D_sample.rows.jsonl)
  - [sample rendered markdown](./2026-04-12_S2D_sample.rendered.md)
- The sample run exposes usable values for `task_id`, repo path, issue title, model/provider, run outcome, indexing status, and snapshot status.

## unsupported_claims

- No automated discovery/backfill command exists yet.
- The current corpus does not emit the target-converged `experiment_id` / `manifest_id` pair.
- Timing, token-cost, and turn-level recovery/misuse fields are still not populated in a way that would support the final longitudinal ledger schema.

## not_checked

- Whether multiple real runs can be merged into a stable append-only JSONL companion without a discovery index.
- Whether the eventual companion should live beside the markdown ledger or remain packet-local until the final manifest format lands.
- Whether the surrogate `task_id`-based row key should be kept, renamed, or mapped to the final key shape later.

## risks

- If the run corpus stays task-id keyed, the final manifest row generator will need a thin normalization layer.
- If the JSONL companion and markdown render are updated independently, the two surfaces can drift.
- If capture remains incomplete for timing or turn-level telemetry, those fields must stay explicitly blocked rather than inferred.

## next_step

- Promote this prototype into a tiny generator only once the formal run manifest emits the canonical key fields, or keep the sample render as the handoff artifact if the corpus stays ad hoc.
