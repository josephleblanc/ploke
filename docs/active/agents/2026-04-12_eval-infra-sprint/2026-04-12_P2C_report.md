## Acceptance Criteria

1. The packet identifies which validity guards are already draft-defined versus missing for formal Phase 2 runs.
2. The packet states one explicit recommendation for the current operational policy: adopt thresholds now, require an EDR first, or keep formal runs blocked pending manifest/config convergence.
3. The packet updates or recommends updates to the live workflow artifacts without overstating what has been formally adopted.

## What Changed

- Reviewed the live workflow/design artifacts that mention validity guards, thresholds, readiness, and formal-run gating.
- Distinguished draft-defined guard fields from actually binding programme policy.
- Accepted a narrow policy decision:
  the current numeric thresholds are real draft examples, but they are not yet globally binding for formal Phase 2 interpretation.
- Seeded `P2D` as the next packet so the first formal baseline/control entry path can adopt one concrete manifest/config surface instead of treating scattered draft artifacts as equivalent.

## Claims

- [1] The workflow already contains draft validity guards for formal experiments, but only in example/config-template form rather than as a globally adopted programme rule.
- [2] The current numeric thresholds are not operationally binding today because the live readiness rule explicitly says they remain draft unless adopted in an EDR or experiment config used for a formal run.
- [3] The right current policy is not "adopt the draft thresholds repo-wide right now"; it is "keep formal baseline/control runs blocked until one concrete formal-run path adopts explicit guards through an EDR or frozen experiment config tied to the manifest path."
- [4] The next blocker after `P2C` is manifest/config convergence, because adoption needs one real formal-run entry surface rather than parallel draft documents that are not yet mapped cleanly onto harness output.
