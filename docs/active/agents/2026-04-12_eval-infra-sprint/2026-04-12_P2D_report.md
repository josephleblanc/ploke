## What Changed

- Compared the draft run-manifest and experiment-config schemas against representative current run artifacts and current `ploke-eval` CLI/runner surfaces.
- Accepted one bounded convergence rule for the first formal Phase 2 packet:
  use a mixed but explicit formal-run entry surface instead of pretending the target-converged manifest already exists.
- Chosen entry surface for the next formal packet:
  `run.json` + `execution-log.json` + `repo-state.json` + `indexing-status.json` + `snapshot-status.json` + `multi-swe-bench-submission.jsonl`, with `record.json.gz` as optional replay-grade support when present.
- Chosen adoption rule:
  validity guards become binding in the first formal-run experiment config, and the paired EDR points at that exact config rather than adopting repo-wide thresholds by implication.
- Seeded `P2E` as the first formal Phase 2 execution-planning packet.

## Claims

- [1] Current run artifacts already support a bounded formal-run entry surface, but they do not support the full target-converged manifest draft as a single emitted object.
- [2] `run.json` is still the primary identity/config anchor, but critical provenance and validity evidence remain split across sidecar files, especially `execution-log.json`, `repo-state.json`, `indexing-status.json`, and `snapshot-status.json`.
- [3] The first formal Phase 2 packet should adopt validity guards in its concrete experiment config and cite them from its EDR, not from the repo-level draft config alone.
- [4] Some draft experiment fields remain doc-only today and must either be waived explicitly for the first formal packet or deferred until harness support exists.

## Recommendation

For the first formal Phase 2 packet:

1. Treat `run.json` as the canonical identity and budget anchor.
2. Treat `execution-log.json` as the canonical model/provider execution source.
3. Treat `repo-state.json`, `indexing-status.json`, and `snapshot-status.json` as the canonical validity/provenance sidecars for checkout, indexing, and snapshot health.
4. Treat `multi-swe-bench-submission.jsonl` as the benchmark-facing output artifact.
5. Treat `record.json.gz` as optional but preferred replay-grade support, not as a hard requirement for the first formal packet.
6. Adopt validity guards in the first formal-run experiment config and point to that config from the packet EDR.
7. Add explicit waivers for draft-only fields that are not yet harness-frozen, especially prompt/tool-schema provenance and any wall-time guard that would otherwise overstate what is captured today.

## Risks

- If the next formal packet treats doc-only config fields as binding without either runtime support or explicit waiver text, the run record will overstate reproducibility.
- The current README surface is narrower than the actual run directory contents, which can cause future workers to miss `record.json.gz` or misread non-canonical files as primary evidence.
- `wall_clock_secs` is still not a trustworthy record-backed summary field, so using wall time as an immediate validity guard or primary comparison metric needs care.
