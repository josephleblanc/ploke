# S2C Metrics Ingestion Bootstrap Report

## implemented

- Added an ingestion/update-path section to [docs/active/workflow/longitudinal-metrics.md](../../workflow/longitudinal-metrics.md).
- Defined a lightweight bootstrap flow that uses the immutable run manifest as the canonical source, stores roll-up rows in an append-only JSONL companion, and regenerates the markdown ledger from that companion.
- Spelled out the operational assumptions and automation boundaries so the design stays honest about what is available now versus later.

## claims

- [AC1] The design now describes a concrete lightest-weight path from new formal run artifacts to the longitudinal metrics surface.
- [AC2] The design proposes a durable machine-readable storage layer: an append-only JSONL companion keyed by `experiment_id` + `manifest_id`.
- [AC3] The design distinguishes what can be automated now from what remains blocked on capture, conventions, or later infrastructure.
- [AC4] The design stays aligned with `eval-design.md` by keeping outcome and validity metrics separate and by treating lower-layer capture as a prerequisite for trustworthy higher-layer interpretation.

## evidence

- Sample source-artifact set to discover from a formal run:
  - `run.json`
  - `execution-log.json`
  - `agent-turn-summary.json`
  - `agent-turn-trace.json`
  - `indexing-status.json`
  - `snapshot-status.json`
  - `repo-state.json`
  - `summary.json` where the experiment workspace writes one
- `docs/workflow/run-manifest.v0.draft.json` defines the target canonical per-run fields that a roll-up row should inherit.
- `docs/workflow/experiment-config.v0.draft.json` shows that formal runs already separate primary outcome metrics from validity guards and secondary metrics.
- `docs/active/workflow/longitudinal-metrics.md` is now the central roll-up surface and includes the proposed ingestion, storage, and regeneration path.
- `docs/active/plans/evals/eval-design.md` defines the outcome-versus-validity framing and the dependency ordering that makes low-level capture the right first automation target.
- `docs/active/workflow/README.md` already treats the longitudinal metrics ledger as a live workflow artifact, so the proposed companion fits the existing workflow layer.

## unsupported_claims

- No actual ingestion script, watcher, or CI hook exists yet.
- The JSONL companion file is a design proposal only; it has not been created or populated.
- I did not verify live experiment directories against the proposed discovery flow.

## not_checked

- Whether every current formal run artifact can already be parsed into the proposed row shape without a small adapter.
- Whether the eventual companion file should live beside the markdown ledger or under a dedicated subdirectory.
- Whether the later automation step should be a local script, a scheduled job, or a repository hook.

## risks

- If formal run naming or artifact layout drifts, discovery can become ambiguous without a small indexing rule.
- If the companion JSONL and markdown ledger are updated independently, the surfaces can diverge.
- If capture for blocked metrics never lands, the ledger will still need explicit missingness handling for those fields.

## next_step

- Implement or prototype a tiny regeneration command or backfill script that emits the JSONL companion from a small sample of formal runs, then regenerate the markdown ledger from that source.
