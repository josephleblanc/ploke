# Review: 21330672 `history: verify sealed artifact at startup`

Verdict: the commit adds a real successor handoff gate. `acknowledge_prototype1_state_handoff` now calls `validate_prototype1_successor_history_admission` before writing successor-ready and before returning `Parent<Ready>` (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3117`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3119`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3130`). That validator requires a present History head, loads the stored sealed block, derives the current checkout tree through `GitWorktreeBackend::clean_tree_key`, and compares it to the sealed artifact claim (`crates/ploke-eval/src/cli/prototype1_process.rs:402`, `crates/ploke-eval/src/cli/prototype1_process.rs:416`, `crates/ploke-eval/src/cli/prototype1_process.rs:419`, `crates/ploke-eval/src/cli/prototype1_process.rs:424`). The invariant is still narrower than several docs/comments imply.

## Findings

1. High: successor admission verifies the current store head, not the predecessor handoff head. The predecessor appends a block and receives `stored_block`, but the successor invocation created immediately afterward does not carry that `BlockHash` or `BlockHead` (`crates/ploke-eval/src/cli/prototype1_process.rs:965`, `crates/ploke-eval/src/cli/prototype1_process.rs:978`; invocation fields are only schema, role, campaign, node, runtime, journal, active root, and timestamp at `crates/ploke-eval/src/cli/prototype1_state/invocation.rs:93`). Startup therefore accepts whichever block is the current lineage head at admission time if that head admits the current tree (`crates/ploke-eval/src/cli/prototype1_process.rs:404`, `crates/ploke-eval/src/cli/prototype1_process.rs:424`). This does gate on sealed History plus checkout tree, but it does not prove "the predecessor's sealed History head" or bind the ready acknowledgement to the exact block sealed before this spawn. Minimal fix: add the appended `BlockHead` or expected `BlockHash` to `SuccessorInvocation`, compare it to the current store head, then verify the sealed block's selected successor/runtime/artifact fields against the invocation before `parent.ready()`.

2. Medium: non-handoff parent startup still bypasses sealed-head admission. If `--handoff-invocation` is absent, `acknowledge_prototype1_state_handoff` returns `parent.ready()` directly (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3065`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3066`), and `Parent<Checked>::ready` is an unchecked cast (`crates/ploke-eval/src/cli/prototype1_state/parent.rs:393`). That may be acceptable for initial/bootstrap runs, but the docs describe startup generally as `Startup<Observed> -> Startup<Validated> -> Parent<Ruling>` and say startup should reject inconsistent surfaces before entering the parent path (`crates/ploke-eval/src/cli/prototype1_state/history.rs:60`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:66`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:671`). Minimal fix: split explicit bootstrap startup from successor startup, and require a checked genesis/absence policy for non-handoff parent starts when a History store exists.

3. Medium: crate-visible construction surfaces still allow future misuse around Crown/History transition mechanics. The raw `Crown<Ruling>` constructor is private, and `seal_block_with_artifact` now requires an admitted artifact claim in the live transition (`crates/ploke-eval/src/cli/prototype1_state/inner.rs:170`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:197`). However, `OpenBlock` authority fields remain `pub(crate)` caller data (`crates/ploke-eval/src/cli/prototype1_state/history.rs:1947`), `SealBlock` fields are also `pub(crate)` and `from_handoff` starts from `Claims::empty_unchecked()` (`crates/ploke-eval/src/cli/prototype1_state/history.rs:1985`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2000`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2010`), and `admit_claim` still accepts the ruler actor identity as data (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2592`). Minimal fix: make block/seal fields private and expose a narrow handoff carrier that derives lineage, predecessor, ruler identity, selected successor, policy, and artifact claim from typed parent/successor inputs.

4. Medium: documentation is stale in both directions after this commit. `history.rs` still says successor startup does not verify the sealed head and is not the live admission gate (`crates/ploke-eval/src/cli/prototype1_state/history.rs:17`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:125`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:273`). `mod.rs` still says live handoff lacks sealed persistence and sealed-head verification (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:194`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:668`). Conversely, `mod.rs` says the single-lineage prototype can directly prevent overlapping mutable Parent authority (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:177`), but there is still no process lease/lock and non-handoff startup can produce another `Parent<Ready>`. Minimal fix: update docs to say "successor handoff now verifies current sealed head artifact tree, but not handoff-bound block identity, process uniqueness, or general bootstrap startup admission."

5. Low: sealed-head loading is intentionally zero-entry only. `sealed_head_block` documents that it rejects non-empty stored blocks until entry loading has its own transition (`crates/ploke-eval/src/cli/prototype1_state/history.rs:698`), and `StoredSealedBlock::into_verified_block` enforces that (`crates/ploke-eval/src/cli/prototype1_state/history.rs:851`). This is compatible with the current minimal handoff blocks, but it will make successor startup fail once live handoff blocks carry admitted entries. Minimal fix: keep this limitation documented or implement verified entry deserialization before adding non-empty live blocks.

## Confirmed Invariants

- The admitted artifact claim is required in the live sealing transition: `seal_block_with_artifact` takes a closure returning `claim::Admitted<Admission, Witnessed<RulerWitness, Verifiable<Artifact, ArtifactLocator>>>` and writes it into `seal.claims` before locking and sealing (`crates/ploke-eval/src/cli/prototype1_state/inner.rs:170`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:217`).
- The startup verifier checks the claim at the right boundary for successor handoff: before successor-ready is recorded and before `Parent<Ready>` is returned (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3117`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3130`).
- `BlockHash` tightening preserves normal storage compatibility: it serializes/deserializes as the same hex string shape while rejecting non-64-hex payloads in memory (`crates/ploke-eval/src/cli/prototype1_state/history.rs:368`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:391`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:400`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:426`). This improves the invariant without changing valid stored hashes.

## Residual Risks

- The store head is still a rebuildable local projection, not an authenticated present/absent proof (`crates/ploke-eval/src/cli/prototype1_state/history.rs:502`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:889`).
- `ArtifactLocator` verifies consistency of a stored `TreeKeyHash` claim; the real environmental observation remains the separate backend `clean_tree_key` call (`crates/ploke-eval/src/cli/prototype1_state/history.rs:1124`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:1193`; `crates/ploke-eval/src/cli/prototype1_process.rs:419`).
- Process uniqueness remains out of scope despite single-ruler wording; the code has no lease/lock/consensus layer, matching the narrower disclaimer in `history.rs` (`crates/ploke-eval/src/cli/prototype1_state/history.rs:121`).
