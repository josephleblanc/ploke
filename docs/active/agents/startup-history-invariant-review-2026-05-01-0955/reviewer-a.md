# Startup History Invariant Review - Reviewer A

## Findings

### 1. High: predecessor startup validates the current artifact/surface, but not that this runtime is the sealed head's selected successor

**Claim:** The new `Startup<Predecessor>` gate is a real improvement over the removed free function, but it still does not fully enforce the documented cross-runtime contract that the successor verifies the predecessor's sealed handoff material before becoming the next parent.

**Evidence:** `acknowledge_prototype1_state_handoff` checks mutable invocation identity and scheduler continuation, then calls `Startup::<Predecessor>::from_history(identity, manifest_path, repo_root)` at [crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3081](../../../../crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3081), [crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3118](../../../../crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3118). The startup constructor loads the current lineage head and verifies the checkout tree and surface against that sealed block at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:481](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:481), [crates/ploke-eval/src/cli/prototype1_state/parent.rs:504](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:504), [crates/ploke-eval/src/cli/prototype1_state/parent.rs:512](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:512), and [crates/ploke-eval/src/cli/prototype1_state/parent.rs:518](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:518). It then only checks lineage and `identity.generation != 0` at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:524](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:524). It does not compare `identity.generation` to `head.block_height() + 1`, does not compare the invocation runtime/node to `SealedBlockHeader::selected_successor`, and does not verify that the parent identity is the successor identity named by the sealed handoff. The sealed header carries successor material at [crates/ploke-eval/src/cli/prototype1_state/history.rs:2511](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:2511), but there is no successor-admission accessor/check used by startup.

**Risk:** A runtime can satisfy "current checkout matches the current sealed artifact claim" without History proving that this exact parent identity/generation/runtime is the immediate selected successor of that head. The missing binding leaves part of the admission contract in invocation JSON and scheduler state rather than in sealed History, which is weaker than the Crown/History model described at [crates/ploke-eval/src/cli/prototype1_state/mod.rs:213](../../../../crates/ploke-eval/src/cli/prototype1_state/mod.rs:213) and [crates/ploke-eval/src/cli/prototype1_state/history.rs:104](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:104).

**Suggested fix:** Add a sealed-head admission method used by `Startup<Predecessor>` that checks successor identity structurally. At minimum it should enforce `identity.generation == head.block_height() + 1`, verify the active parent identity/node against the sealed selected successor or a sealed successor-selection claim, and bind the invocation runtime id to the sealed handoff evidence when an invocation is present.

### 2. Medium: genesis `Startup<Validated>` is local absence plus generation zero, not the documented artifact/surface admission base case

**Claim:** The genesis path now distinguishes itself from predecessor admission, but it remains weaker than the docs' "lineage/artifact" bootstrap language.

**Evidence:** The no-handoff path in `acknowledge_prototype1_state_handoff` unconditionally chooses `Startup::<Genesis>::from_history` at [crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3064](../../../../crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3064). That constructor reads `lineage_state` and accepts `StoreHead::Absent` for generation 0 at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:442](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:442), [crates/ploke-eval/src/cli/prototype1_state/parent.rs:459](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:459), and [crates/ploke-eval/src/cli/prototype1_state/parent.rs:468](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:468). It does not compute or bind the clean tree key, surface commitment, or parent identity evidence into a genesis History epoch. The docs say genesis is a configured-store absence claim for the lineage/artifact at [crates/ploke-eval/src/cli/prototype1_state/mod.rs:93](../../../../crates/ploke-eval/src/cli/prototype1_state/mod.rs:93) and that unreadable, ambiguous, or checkout-inconsistent stores must reject at [crates/ploke-eval/src/cli/prototype1_state/history.rs:98](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:98). The current docs also acknowledge that setup does not open or append a genesis block at [crates/ploke-eval/src/cli/prototype1_state/history.rs:351](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:351).

**Risk:** A gen0 parent can enter `Parent<Ready>` before there is a sealed genesis authority epoch for the artifact/surface it is running from. The single-ruler History model therefore begins at first handoff, not at initial parent readiness.

**Suggested fix:** Either open/append a genesis block before `Parent<Ready>`, or narrow the new `Startup<Genesis>` docs to say it is only local lineage-head absence plus the existing `Parent<Checked>` checkout validation. If preserving the stronger claim, compute and store the genesis tree key, surface commitment, and parent identity evidence during startup.

### 3. Medium: `Startup<Validated>` can become stale before `Parent<Ready>`, despite diagnostics implying head-change protection

**Claim:** `Startup<Validated>` records the head observed during validation, but `Parent<Checked>::ready` does not re-read the History store before accepting it.

**Evidence:** `Startup::<Predecessor>::from_history` reads the current `LineageState` once at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:487](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:487) and returns a self-contained `Startup<Validated>` at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:521](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:521). `Parent<Checked>::ready` only calls `startup.validate_parent(&self.identity)` at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:432](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:432). `validate_parent` compares the stored `StartupKind::Predecessor { head }` to the stored `LineageState` at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:553](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:553), but it does not load the current store head. The error text "head changed after validation" at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:573](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:573) can only detect internal inconsistency in the carried value, not an actual store change after validation.

**Risk:** Two local processes can both validate against the same head and enter ready/ruling preparation if no external lease or re-check exists. Append-time stale-state checks still protect block append, but the startup gate itself is weaker than the "single-ruler local policy" wording suggests.

**Suggested fix:** Make `Parent<Checked>::ready` or `Startup<Validated>::validate_parent` re-read the store and reject if the current head/root differs from the validated state. Longer term, pair this with the documented lease/lock mechanism for process uniqueness.

### 4. Medium: documentation is partly stale after this diff and partly overstates what the new carrier proves

**Claim:** The docs now mix three statuses: implemented carrier, partially implemented startup admission, and intended full `ProducedBy`/`AdmittedBy` proof.

**Evidence:** `history.rs` says bootstrap admission and a uniform `Startup<Validated>` carrier remain incomplete at [crates/ploke-eval/src/cli/prototype1_state/history.rs:23](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:23), and later says the implementation does not yet enforce a uniform bootstrap/predecessor admission carrier at [crates/ploke-eval/src/cli/prototype1_state/history.rs:249](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:249). This diff adds a `Startup<Validated>` carrier for the `prototype1-state` parent-ready path at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:72](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:72) and wires it at [crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3057](../../../../crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3057). Conversely, `mod.rs` now states that `Parent<Ready>` requires `Startup<Validated>` checking either genesis absence or predecessor head/artifact/surface at [crates/ploke-eval/src/cli/prototype1_state/mod.rs:419](../../../../crates/ploke-eval/src/cli/prototype1_state/mod.rs:419), but the full intended `ProducedBy(SelfRuntime, CurrentArtifact)` and `AdmittedBy(CurrentArtifact, Lineage, Policy, History)` procedure remains only partially implemented as described at [crates/ploke-eval/src/cli/prototype1_state/history.rs:70](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:70).

**Risk:** Future readers can over-trust `Startup<Validated>` as the complete startup admission proof, or under-trust it as still absent. Both readings are wrong: the carrier exists for the live parent-ready path, but the predecessor identity/generation binding and genesis artifact/surface base case are still incomplete.

**Suggested fix:** Update the status notes in `history.rs` and `mod.rs` to distinguish "carrier wired for `prototype1-state` readiness" from "full startup admission proof implemented." Keep explicit caveats for missing successor identity binding, stale-head re-check, genesis block creation, and process uniqueness.

### 5. Low: `Startup<Validated>` erases the admission path into an internal enum, which is acceptable locally but is a structural smell for future protocol checks

**Claim:** The new names are much better than the removed `validate_prototype1_successor_history_admission` free function, but the richer object is not just "validated startup"; it is validated startup by a particular admission basis.

**Evidence:** `Startup::<Genesis>::from_history` and `Startup::<Predecessor>::from_history` both return `Startup<Validated>` at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:446](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:446) and [crates/ploke-eval/src/cli/prototype1_state/parent.rs:486](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:486). The basis is then stored as `StartupKind` at [crates/ploke-eval/src/cli/prototype1_state/parent.rs:89](../../../../crates/ploke-eval/src/cli/prototype1_state/parent.rs:89). That duplication of a type-state path plus runtime enum is manageable now, but it makes it easier to add checks to the wrong layer as startup admission grows.

**Risk:** Follow-on work may keep adding flags and helper names around startup instead of modeling the basis as structured evidence. This is the same class of drift AGENTS.md warns about: role/state meaning starts living in names and side fields rather than carriers.

**Suggested fix:** If successor identity, genesis surface evidence, and upgrade/fork policy are added, consider preserving the basis structurally, for example with a typed admission evidence value consumed into `Startup<Validated<Genesis>>` / `Startup<Validated<Predecessor>>`, or with a `ValidatedStartup { basis: GenesisEvidence | PredecessorEvidence }` whose variants carry the exact checked evidence.

## Non-Issues / Strengths

- `Parent<Checked>::ready` now requires a `Startup<Validated>` argument, so the main live path can no longer advance from checked to ready by a trivial public status write.
- `Startup<S>` fields and `Startup::validated` are private to `parent.rs`, so sibling modules cannot directly mint `Startup<Validated>` despite the marker types being `pub(crate)`.
- `Crown<Ruling>` and `Crown<Locked>` remain protected by private fields and private state-marker constructors in `inner.rs`; sibling modules cannot mint raw ruling or locked Crowns from a lineage string.
- `Block<Open>::open` and `Block<Open>::seal` remain private to `history.rs`; crate-visible block opening/sealing still routes through Crown methods.
- `FsBlockStore::append` still consumes the expected `LineageState` and verifies the opened-from state root before advancing the local head.
- `cargo check -p ploke-eval --tests` passed. It emitted existing dead-code/unused warnings, but no compile errors from the transition signature change.

## Verdict

Commit with caveats. The diff is directionally correct and improves the live transition surface, but it should not be treated as the full Crown/History startup invariant. Before relying on it for longer runs, tighten predecessor identity/generation binding and update the docs so `Startup<Validated>` is described as the current partial carrier, not the complete admission proof.
