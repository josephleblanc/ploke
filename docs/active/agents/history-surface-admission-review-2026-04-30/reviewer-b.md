# Reviewer B: History Surface Admission

Reviewed commit: `2ba2f54e history: enforce surface admission boundaries`

Scope: last commit only, with emphasis on Prototype 1 History/Crown/surface admission claims, type barriers, and naming/structure.

## Findings

### High: child invocation execution is not protected by the surface gate

Invariant claim: the docs state the inductive ordinary-succession rule as "a Parent whose Artifact has digest D may execute a child/successor runtime only after proving the child/successor Artifact also has digest D" and conclude that every executed descendant produced by the transition system has digest D ([docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:174](../../../../docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:174), [docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:181](../../../../docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:181), [docs/workflow/evalnomicon/drafts/history-blocks-and-crown-authority.md:74](../../../../docs/workflow/evalnomicon/drafts/history-blocks-and-crown-authority.md:74)). The same docs specifically claim child evaluation validates the partition before child build/hydration and again after persistence ([docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:221](../../../../docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:221), [docs/workflow/evalnomicon/drafts/history-blocks-and-crown-authority.md:91](../../../../docs/workflow/evalnomicon/drafts/history-blocks-and-crown-authority.md:91)).

What code enforces: the parent-side paths do call `validate_child_surface` before build and after persistence ([crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3301](../../../../crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3301), [crates/ploke-eval/src/cli/prototype1_process.rs:798](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:798), [crates/ploke-eval/src/cli/prototype1_process.rs:1839](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:1839), [crates/ploke-eval/src/cli/prototype1_process.rs:1855](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:1855)). But the actual child process entrypoint `execute_prototype1_runner_invocation` loads the invocation, node, and request, marks the node running, records child-ready, and runs evaluation without recomputing or checking the surface ([crates/ploke-eval/src/cli/prototype1_process.rs:1669](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:1669), [crates/ploke-eval/src/cli/prototype1_process.rs:1687](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:1687), [crates/ploke-eval/src/cli/prototype1_process.rs:1700](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:1700), [crates/ploke-eval/src/cli/prototype1_process.rs:1720](../../../../crates/ploke-eval/src/cli/prototype1_process.rs:1720)). The CLI exposes this direct executable path for child invocations ([crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:1316](../../../../crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:1316)).

Why this matters: the validated surface is not bound to the runtime boundary. There is a caller-discipline window between parent validation and child execution, and direct invocation execution can run from the invocation/request workspace without a fresh surface check. That weakens the documented "before executing candidate runtime" invariant.

Patch to close: put the expected surface commitment, or at least the active parent root plus expected immutable root, into the child invocation/box. At child startup, before `update_node_status` or child-ready, recompute `GitWorktreeBackend::surface_commitment(active_parent_root, workspace_root)` and reject on mismatch. The transition should produce a typed `Child<SurfaceChecked>` or equivalent carrier before `Child<Evaluating>`.

### Medium: surface commitments can be fabricated inside the crate

Invariant claim: surface admission is supposed to be a static reconstruction witness computed from checked-out Artifacts, not an arbitrary digest/status object ([docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:213](../../../../docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:213)). The implementation docs echo that the surface commitment is "computed statically from Artifacts without running the candidate Runtime" ([crates/ploke-eval/src/cli/prototype1_state/history.rs:2426](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:2426)).

What code enforces: the live backend does compute roots from git-tracked `crates/ploke-eval` files and tool-description files, and rejects immutable changes ([crates/ploke-eval/src/cli/prototype1_state/backend.rs:1164](../../../../crates/ploke-eval/src/cli/prototype1_state/backend.rs:1164), [crates/ploke-eval/src/cli/prototype1_state/backend.rs:1185](../../../../crates/ploke-eval/src/cli/prototype1_state/backend.rs:1185), [crates/ploke-eval/src/cli/prototype1_state/backend.rs:1192](../../../../crates/ploke-eval/src/cli/prototype1_state/backend.rs:1192)). However, `SurfaceRoot::new`, `Surface::new`, `SurfaceDelta::new`, and `SurfaceCommitment::new` are all `pub(crate)` constructors over caller-supplied roots ([crates/ploke-eval/src/cli/prototype1_state/history.rs:1152](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:1152), [crates/ploke-eval/src/cli/prototype1_state/history.rs:1171](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:1171), [crates/ploke-eval/src/cli/prototype1_state/history.rs:1196](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:1196), [crates/ploke-eval/src/cli/prototype1_state/history.rs:1221](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:1221)). `OpenBlock` also accepts a caller-supplied `SurfaceCommitment` as data ([crates/ploke-eval/src/cli/prototype1_state/history.rs:2152](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:2152), [crates/ploke-eval/src/cli/prototype1_state/history.rs:2162](../../../../crates/ploke-eval/src/cli/prototype1_state/history.rs:2162)).

Why this matters: the current live path uses the backend correctly, but the type barrier does not encode "backend-derived from Artifacts." Any sibling module can mint a surface commitment that passes `verify_current` if it chooses matching hashes. This leaves the admission proof partly as caller discipline.

Patch to close: make raw surface constructors private or `pub(super)` to `history` tests only, and expose a crate-visible backend/admission constructor that takes checked Artifact roots or a backend-produced capability. Consider a typed carrier such as `Surface<Checked>` or `ArtifactSurface<Derived>` so `OpenBlock` cannot receive a raw commitment assembled from arbitrary hashes.

## Naming And Structure Audit

- `validate_child_surface` is mechanism-shaped and returns a raw `SurfaceCommitment`; it does not produce a role/state carrier. A stronger structure would name the transition result, for example `Child<SurfaceChecked>`, and make evaluation require that state.
- `validate_prototype1_successor_history_admission` implies admission is completed by a function call, but it returns `()` and does not create a typed `Startup<Validated>` or `Parent<Ruling>` boundary. This is acceptable as a staged bridge, but the name is stronger than the type barrier.
- `SurfaceCommitment` is a reasonable structural carrier for `Immutable + Mutated + Ambient`; it avoids flattened names such as `immutable_surface_root_before`. The weak point is constructor visibility, not the carrier shape.
