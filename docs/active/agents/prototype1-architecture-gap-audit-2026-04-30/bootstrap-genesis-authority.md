# Bootstrap / Genesis Authority Audit

Date: 2026-04-30

Focus: how Prototype 1 admits the first Parent, what is special-cased for gen0/bootstrap, and whether the current code has a clear genesis authority model or only an intended one.

## Finding

The clearer genesis authority model exists, but it is only partially implemented. The code has a typed `GenesisAuthority` value, an `OpeningAuthority::Genesis` block header case, a `StoreHead::Absent` append check for height 0, and successor-side sealed-head validation. It does not yet have a uniform startup admission carrier that gates the first Parent before that Parent enters the live loop.

In the current live path, the first Parent is admitted by setup/caller discipline:

- `prototype1-setup` registers a root parent node, checks out a fresh `prototype1-parent-<campaign>-gen0` branch, writes parent identity, commits it, and validates the parent checkout, but it does not open or append a genesis History block (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:153`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:181`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:190`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:197`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:203`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:211`).
- The alternate `prototype1-state --init-parent-identity` path requires a generation-0 node and fresh branch, writes/commits identity, and validates the checkout, but also returns before normal parent execution and does not touch History (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2988`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2994`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3003`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3012`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3018`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3024`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3032`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3186`).
- Normal `prototype1-state` resolves an existing parent identity, checks checkout/node/selection consistency, and only validates History admission when a successor handoff invocation is supplied (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3217`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:334`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:347`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:353`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:363`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3058`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3066`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3118`).

The first actual History block is created later, during successor handoff. If the configured `FsBlockStore` has no head for the campaign lineage, `handoff_block_fields` creates block height 0 with `OpeningAuthority::Genesis(GenesisAuthority::new(...))`; this is appended before the successor process is spawned (`crates/ploke-eval/src/cli/prototype1_process.rs:974`, `crates/ploke-eval/src/cli/prototype1_process.rs:991`, `crates/ploke-eval/src/cli/prototype1_process.rs:1177`, `crates/ploke-eval/src/cli/prototype1_process.rs:1187`, `crates/ploke-eval/src/cli/prototype1_process.rs:1196`, `crates/ploke-eval/src/cli/prototype1_process.rs:1211`, `crates/ploke-eval/src/cli/prototype1_process.rs:1010`, `crates/ploke-eval/src/cli/prototype1_process.rs:1033`).

## Implemented Invariants

- Sealed blocks, once constructed, are locally tamper-evident: sealing computes entry root and block hash from header/entries, and verification recomputes them (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2570`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2576`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2588`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2647`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2667`).
- `Block<Open>` construction is private and routed through `Crown<Ruling>::open_block`, which checks the Crown lineage against the requested block lineage (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2477`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2768`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2778`).
- `Block<Open>::seal` is private and crate-visible sealing is routed through `Crown<Locked>::seal`, which checks lineage before sealing (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2570`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2872`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2878`).
- The live handoff path retires `Parent<Selectable>`, creates a private `Crown<Ruling>`, opens the block, admits the artifact claim, locks the Crown, and seals the block in one move-only path (`crates/ploke-eval/src/cli/prototype1_state/inner.rs:188`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:197`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:214`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:216`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:217`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:218`).
- The store append boundary checks expected head freshness and enforces absent-head genesis shape: absent head accepts only height 0 with no parents; present head rejects duplicate genesis and requires consecutive height and parent hash linkage (`crates/ploke-eval/src/cli/prototype1_state/history.rs:793`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:801`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:808`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:985`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:994`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:1008`).
- Store-head absence is not a blind missing-file check. Missing `heads.json` is allowed only when there are no stored block/index records; otherwise missing projections error (`crates/ploke-eval/src/cli/prototype1_state/history.rs:669`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:671`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:674`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:684`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:840`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:842`).
- Successor startup now rejects absent History head, loads/verifies the sealed head, recomputes the current clean tree key and surface commitment, and checks both against the sealed block (`crates/ploke-eval/src/cli/prototype1_process.rs:397`, `crates/ploke-eval/src/cli/prototype1_process.rs:404`, `crates/ploke-eval/src/cli/prototype1_process.rs:416`, `crates/ploke-eval/src/cli/prototype1_process.rs:419`, `crates/ploke-eval/src/cli/prototype1_process.rs:424`, `crates/ploke-eval/src/cli/prototype1_process.rs:427`, `crates/ploke-eval/src/cli/prototype1_process.rs:430`).

## Aspirational Or Deferred Invariants

- The module docs explicitly say startup should become `Startup<Observed> -> Startup<Genesis>|Startup<Predecessor> -> Startup<Validated> -> Parent<Ruling>`, but that is not the live startup type path (`crates/ploke-eval/src/cli/prototype1_state/history.rs:65`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:71`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:73`).
- The top-level Prototype 1 state docs describe the intended admission check as `ProducedBy(SelfRuntime, CurrentArtifact)` plus `AdmittedBy(CurrentArtifact, Lineage, Policy, History)`, and explicitly note that bootstrap startup is still local configured-store absence rather than a uniform admission carrier (`crates/ploke-eval/src/cli/prototype1_state/mod.rs:85`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:93`, `crates/ploke-eval/src/cli/prototype1_state/mod.rs:97`).
- The intended genesis rule is local, configured-store absence for lineage/artifact, and unreadable/ambiguous/inconsistent stores should reject rather than silently bootstrap (`crates/ploke-eval/src/cli/prototype1_state/history.rs:93`). The store projection checks implement some local ambiguity rejection, but there is no startup carrier that binds absence, current artifact, policy surface, and first Parent before the first Parent turn.
- Docs state the first Parent should be admitted with policy-surface digest `D`, making later descendant execution inductive over that digest (`crates/ploke-eval/src/cli/prototype1_state/history.rs:141`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:148`; `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:174`). The current concrete surface commitment exists for handoff/successor validation, but the base case is not established before gen0 runs.
- `history.rs` calls out the remaining gaps directly: bootstrap admission, full authenticated state roots, policy-surface digest checks, and process uniqueness are not implemented (`crates/ploke-eval/src/cli/prototype1_state/history.rs:19`). It also states that gen0 setup writes and commits parent identity but does not open or append a genesis History block; the first live handoff creates genesis if no configured head exists (`crates/ploke-eval/src/cli/prototype1_state/history.rs:344`).
- The conceptual docs say the current code lacks a uniform bootstrap/predecessor startup admission carrier, and bootstrap/non-handoff startup does not yet have the same admission shape as successor startup (`docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:39`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:83`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:95`).

## Ambiguous Claims

- `GenesisAuthority` looks like a structural model, but its constructor is just `pub(crate) fn new(...)` taking `ProcedureRef`, `TreeKeyHash`, and `ParentIdentityRef`; it records bootstrap material and explicitly says it is not a global absence proof (`crates/ploke-eval/src/cli/prototype1_state/history.rs:1804`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:1811`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:1817`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:1823`). The actual absence evidence is supplied outside the type by `store.head(...)` in `handoff_block_fields` and checked again by `append(...)`.
- `OpenBlock` has public crate fields and is assembled in `prototype1_process.rs`; its docs explicitly say block opening still receives validated material from the current process boundary and lacks a uniform typed startup/admission carrier (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2160`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2167`; `crates/ploke-eval/src/cli/prototype1_process.rs:1210`).
- Crown lineage is currently a debug string derived from `ParentIdentity.campaign_id`, not a fully modeled lineage coordinate (`crates/ploke-eval/src/cli/prototype1_state/inner.rs:91`, `crates/ploke-eval/src/cli/prototype1_state/inner.rs:108`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:423`). This is a useful barrier against accidental cross-lineage sealing, but not a complete authority model.
- `Crown<Ruling>::admit_claim` proves possession of the Crown but still accepts the ruler actor identity as caller data. The implementation notes this gap and expects a future `Parent<Ruling>` carrier to supply identity structurally (`crates/ploke-eval/src/cli/prototype1_state/history.rs:2831`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2839`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2844`).
- The docs sometimes phrase “the configured first Parent is admitted with policy-surface digest D” as the base case for the inductive invariant (`docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:177`). Current code does not yet perform that base admission at first-parent startup; it only begins the History chain when the first parent reaches successor handoff.

## Bootstrap / Gen0 Special Cases

- Gen0 identity creation is intentionally restricted to generation 0 and a fresh identity branch, but that is identity/setup validation, not History admission (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2994`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2995`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3003`).
- The first History block is special-cased on `StoreHead::Absent` during handoff: height 0, no parent hashes, and `OpeningAuthority::Genesis(...)` (`crates/ploke-eval/src/cli/prototype1_process.rs:1187`, `crates/ploke-eval/src/cli/prototype1_process.rs:1196`).
- Successor startup is not allowed to use that absent-head case; it requires an existing sealed head and errors if none is present (`crates/ploke-eval/src/cli/prototype1_process.rs:404`, `crates/ploke-eval/src/cli/prototype1_process.rs:406`).

## Loop Completion Expectation

I expect the full single-successor loop can complete with the current implementation, assuming the ordinary build/eval/scheduler/git operations succeed. Bootstrap/genesis handling should not by itself prevent the first generation from running or handing off, because the first Parent can start from setup identity without History admission, and the first handoff will create and append the genesis block when the configured store head is absent.

That completion is weaker than the documented startup invariant. The first Parent's own entry into the loop is not proven by a sealed History head or a `Startup<Genesis> -> Startup<Validated>` transition. The first History block is a retrospective genesis authority epoch produced at the first handoff, not a prerequisite for the gen0 Parent to rule. After that first handoff, the successor path is substantially stronger: it has a sealed head, tree-key claim, and surface commitment to validate before entering the typed parent run.

The practical consequence is that a successful run demonstrates that the current local trampoline can bootstrap and continue. It does not yet demonstrate the stronger claim that every Parent, including the first one, entered `Parent<Ruling>` through the same History/Crown admission model.
