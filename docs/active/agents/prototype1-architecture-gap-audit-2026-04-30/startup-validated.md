# Startup Validated Gate Audit

Date: 2026-04-30

Audit focus: whether Prototype 1 currently has a universal `Startup<Validated>`-style gate before a runtime can enter meaningful Parent/Child/Ruler paths.

## Summary

Prototype 1 partially implements the startup gate, but not universally and not as `Startup<Validated>` or an equivalent structural carrier.

The live successor handoff path now has a real predecessor-admission gate: a successor invocation validates scheduler continuation, rejects missing sealed History head, recomputes the active checkout's clean tree key, verifies that tree against the sealed head's admitted Artifact claim, recomputes the current surface, and checks it against the sealed head before `Parent<Checked>` becomes `Parent<Ready>` (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3118`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3119`, `crates/ploke-eval/src/cli/prototype1_process.rs:397`, `crates/ploke-eval/src/cli/prototype1_process.rs:404`, `crates/ploke-eval/src/cli/prototype1_process.rs:419`, `crates/ploke-eval/src/cli/prototype1_process.rs:427`, `crates/ploke-eval/src/cli/prototype1_process.rs:430`).

The non-handoff/bootstrap path does not have the same gate. If no `--handoff-invocation` is supplied, `acknowledge_prototype1_state_handoff` returns `parent.ready()` directly after checkout/identity validation (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3064`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3065`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3066`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3067`). That means the architecture does not currently enforce "every runtime must pass startup admission before entering the parent path." It enforces "successors entering through the live handoff invocation must pass sealed-head admission."

## Ambiguous Or Over-Strong Claims

The phrase "A Runtime may enter `Parent<Ruling>` only after establishing..." in the v2 Startup section reads as the target invariant, but it could be misread as already universal unless kept with the surrounding status note (`docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:85`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:88`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:91`, `docs/workflow/evalnomicon/chat-history/history-blocks-v2.md:92`). The docs and code use "policy-surface digest" language, but the implemented structural object is `SurfaceCommitment` over hardcoded partitions, not a general protocol-upgrade or policy system. ## Does The Architecture Already Handle This Under Another Name?

For successor handoff only, yes: the current name is `validate_prototype1_successor_history_admission`, with supporting sealed-head methods `verify_current_artifact_tree` and `verify_current_surface` (`crates/ploke-eval/src/cli/prototype1_process.rs:397`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2702`, `crates/ploke-eval/src/cli/prototype1_state/history.rs:2736`). This is the live predecessor-admission half of `Startup<Validated>`.

For the full architecture, no. There is no structural `Startup<Observed>`, `Startup<Genesis>`, `Startup<Predecessor>`, or `Startup<Validated>` carrier. There is also no transition of the form `Parent<Checked> + Startup<Validated> -> Parent<Ready>`. The existing `Parent<Checked>::ready(self) -> Parent<Ready>` takes no admission evidence (`crates/ploke-eval/src/cli/prototype1_state/parent.rs:388`, `crates/ploke-eval/src/cli/prototype1_state/parent.rs:393`). The call site conditionally performs successor admission by function-call sequencing, then calls `parent.ready()` (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3119`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3131`), while the no-handoff branch calls `parent.ready()` without History admission (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3066`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3067`).

## Expected Loop Completion

I expect the current bounded loop structure can complete in the ordinary handoff path, assuming the active checkout is clean, the generated child builds, the continuation decision is `ContinueReady`, the successor process starts within the timeout, and the local History store projections are consistent. The implementation seals and appends a History block before successor spawn (`crates/ploke-eval/src/cli/prototype1_process.rs:1010`, `crates/ploke-eval/src/cli/prototype1_process.rs:1033`, `crates/ploke-eval/src/cli/prototype1_process.rs:1035`, `crates/ploke-eval/src/cli/prototype1_process.rs:1080`), and the successor validates that sealed head before entering the typed parent turn (`crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3119`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3129`, `crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:3261`).

The missing universal startup gate should not by itself prevent a normal first handoff and successor generation from completing. The first parent can start without a History admission carrier, then the first handoff creates the genesis block if the store is absent (`crates/ploke-eval/src/cli/prototype1_process.rs:1196`, `crates/ploke-eval/src/cli/prototype1_process.rs:1199`). After that, handoff-launched successors use the sealed-head gate.

The gap matters for correctness and restart safety, not just aesthetics. A non-handoff `prototype1-state` invocation can still become `Parent<Ready>` from checkout/identity/scheduler discipline alone, even if a sealed head exists and the active checkout does not match it. That path can enter meaningful parent work without proving `AdmittedBy(CurrentArtifact, Lineage, Policy, History)`.

## Conclusion

Current state: partial implementation.

Implemented: successor handoff has a concrete sealed-head Artifact and surface gate, Crown/block construction is substantially type-mediated, and sealed block appends check local lineage-head consistency.

Not implemented: a universal `Startup<Validated>`-style carrier that all Parent startup paths must consume before entering `Parent<Ready>` or any future `Parent<Ruling>` path.

The right architecture is already documented: split startup into bootstrap absence and predecessor sealed-head cases, produce a typed validated startup carrier, and require that carrier for the transition into the authoritative parent path. Until then, the accurate claim is: "live handoff-launched successors validate against sealed History before entering the typed parent path"; not "every runtime must pass a universal startup gate before becoming meaningful."
