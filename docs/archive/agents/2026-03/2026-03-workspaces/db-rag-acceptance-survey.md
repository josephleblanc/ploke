# DB/RAG Workspace Acceptance Survey

Survey basis: direct code inspection only. This doc focuses on the current search/index contracts in `ploke-db` and `ploke-rag`, plus the workspace/namespace hooks that already exist but are not yet threaded through retrieval.

## Summary

The current search stack is still crate-scoped by construction. Dense search is bound to one active embedding set, BM25 is one in-memory actor rebuilt from the whole database, and `RagService` exposes no crate/workspace scope parameter on any retrieval entrypoint. Workspace/namespace data does exist in the DB layer, but it is mostly used for identification, restore, and path disambiguation rather than query filtering.

## Current Search Contracts

- `RagService` exposes `search_bm25`, `search_bm25_strict`, `search`, `hybrid_search`, and `get_context`, but none accept a workspace or crate scope. BM25 has explicit fallback/retry semantics, while dense search fans out across all primary node types and applies `top_k` after merging results. See [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L234), [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L424), [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L489), [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L656), [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L550).
- `search` in `RagService` calls `search_similar_args`, which immediately captures the current active embedding set from `Database`; there is no workspace slice or namespace filter in the call chain. See [`crates/ploke-db/src/index/hnsw.rs`](crates/ploke-db/src/index/hnsw.rs#L102), [`crates/ploke-db/src/index/hnsw.rs`](crates/ploke-db/src/index/hnsw.rs#L118), [`crates/ploke-db/src/index/hnsw.rs`](crates/ploke-db/src/index/hnsw.rs#L75).
- Dense DB search is set-specific rather than workspace-specific: `search_similar_for_set` runs against one `EmbeddingSet`, filters only by that set id, and projects `namespace` in the output rows. The namespace is preserved, but not used to constrain search. See [`crates/ploke-db/src/multi_embedding/hnsw_ext.rs`](crates/ploke-db/src/multi_embedding/hnsw_ext.rs#L337), [`crates/ploke-db/src/multi_embedding/hnsw_ext.rs`](crates/ploke-db/src/multi_embedding/hnsw_ext.rs#L367), [`crates/ploke-db/src/multi_embedding/hnsw_ext.rs`](crates/ploke-db/src/multi_embedding/hnsw_ext.rs#L390).
- BM25 rebuild/search is also whole-DB and not scoped. `collect_rebuild_sources` walks every primary node relation, `rebuild_from_db` builds a single index over that corpus, and the BM25 actor’s `Search` command only carries `query` and `top_k`. See [`crates/ploke-db/src/bm25_index/mod.rs`](crates/ploke-db/src/bm25_index/mod.rs#L715), [`crates/ploke-db/src/bm25_index/mod.rs`](crates/ploke-db/src/bm25_index/mod.rs#L518), [`crates/ploke-db/src/bm25_index/bm25_service.rs`](crates/ploke-db/src/bm25_index/bm25_service.rs#L19).
- `context::assemble_context` consumes ranked IDs and then looks up node metadata/snippets; it assumes retrieval already chose the correct scope. See [`crates/ploke-rag/src/context/mod.rs`](crates/ploke-rag/src/context/mod.rs#L164), [`crates/ploke-rag/src/context/mod.rs`](crates/ploke-rag/src/context/mod.rs#L185).

## Existing Namespace/Workspace Support

- `ploke-core` already has first-class `CrateId`, `WorkspaceId`, `CrateInfo`, and `WorkspaceInfo` derived from canonical root paths, so workspace identity exists even though RAG does not consume it yet. See [`crates/ploke-core/src/workspace.rs`](crates/ploke-core/src/workspace.rs#L7), [`crates/ploke-core/src/workspace.rs`](crates/ploke-core/src/workspace.rs#L72).
- DB queries already carry `namespace` through file/module resolution and search result projections. `get_crate_files` returns namespace-bearing file rows, `get_nodes_ordered` and related lookup paths include namespace, and exact resolvers preserve namespace in the output shape. See [`crates/ploke-db/src/database.rs`](crates/ploke-db/src/database.rs#L390), [`crates/ploke-db/src/database.rs`](crates/ploke-db/src/database.rs#L1402), [`crates/ploke-db/src/helpers.rs`](crates/ploke-db/src/helpers.rs#L20), [`crates/ploke-db/src/helpers.rs`](crates/ploke-db/src/helpers.rs#L80).
- Active embedding metadata is already persisted per crate name and restored preferentially from metadata after backup import. See [`crates/ploke-db/src/database.rs`](crates/ploke-db/src/database.rs#L827), [`crates/ploke-db/src/database.rs`](crates/ploke-db/src/database.rs#L903), [`crates/ploke-db/src/database.rs`](crates/ploke-db/src/database.rs#L976).

## Invariants And Likely Failure States

- Dense search depends on the current active embedding set and the registered HNSW index for that set; a missing index currently surfaces as a warning path, not as a scoped fallback. See [`crates/ploke-db/src/multi_embedding/hnsw_ext.rs`](crates/ploke-db/src/multi_embedding/hnsw_ext.rs#L448), [`crates/ploke-db/src/index/hnsw.rs`](crates/ploke-db/src/index/hnsw.rs#L75).
- `search_bm25` can fall back to dense search when BM25 is uninitialized, building, or empty; strict mode instead returns an error on those states. That fallback behavior is part of the current contract and should not be changed implicitly by workspace scoping. See [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L296), [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L310), [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L471).
- `bm25_load` currently ignores the sidecar path as a source of truth and rebuilds from the DB instead. That is a deliberate failure-tolerant behavior, but it means BM25 state is not yet a workspace-scoped artifact. See [`crates/ploke-db/src/bm25_index/bm25_service.rs`](crates/ploke-db/src/bm25_index/bm25_service.rs#L161).
- Workspace restore is still keyed by crate-name prefix and active-set metadata; if the backup file cannot be found or the metadata is missing, restore can fail or fall back to the first populated set. See [`crates/ploke-db/src/multi_embedding/db_ext.rs`](crates/ploke-db/src/multi_embedding/db_ext.rs#L1169), [`crates/ploke-db/src/multi_embedding/db_ext.rs`](crates/ploke-db/src/multi_embedding/db_ext.rs#L1195), [`crates/ploke-db/src/database.rs`](crates/ploke-db/src/database.rs#L1001).
- `RagService::search` hardcodes a broad radius when it constructs `SimilarArgs`, so the current dense path does not yet expose a workspace or relevance-scope knob beyond `top_k` and per-type `ef`/`max_hits`. See [`crates/ploke-rag/src/core/mod.rs`](crates/ploke-rag/src/core/mod.rs#L516).

## Candidate Acceptance Criteria

- A loaded workspace can be searched without leaking hits from crates outside that workspace.
- Dense, BM25, and hybrid retrieval all accept the same scope model and return the same crate/workspace slice for the same query.
- Optional crate restriction narrows search to one crate namespace without changing ranking semantics inside that namespace.
- Context assembly only materializes snippets for IDs that already passed the workspace/crate scope filter.
- Workspace-aware retrieval preserves current failure semantics for unready BM25 and missing HNSW indexes.
- Search tests can prove scope by checking returned IDs against `namespace` or crate membership, not just by matching snippet text.
- Workspace restore/load tests can prove that the active embedding set and indexed scope are restored together, not independently.
