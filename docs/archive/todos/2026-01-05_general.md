## Workflow Issues

### Poor discovery loop issues

While we don't have a formal way to distinguish different phases of model behavior right now (and don't want to add state tracking around this yet), in what I'll call the "context gathering" stage the model seems to be over-gathering context.

Typically, the user will ask some question related to the code base. Our internal RAG system then generates a `ContextPlan` from a dense + (optionally) sparse search of the code base for code items related to the user's query. What usually happens next is that the model begins searching for related code items.

However, rather than using the `request_code_context` tool, the LLMs seem to uniformly default to the `read_file` tool (alias of `ns_read`). This sidesteps a lot of the benefits provided by our application, which hopes to make the code search more targeted by having the model's search go through either the dense vector search or bm25 dense embedding search. It would be better if we could expose the search capabilities of our application in more clear ways that the model tended to use more often, or added instructions or guardrails so the model would use these code search capabilities.

Possible solutions:

1. Add instructions to the system messages providing the LLM with more directed instructions to use the code search tools, possibly as part of a kind of "workflow instructions", e.g.

"When the user asks you to search the code base, or asks a question that leads you to search the code base, **ALWAYS** try to use the code search tool `request_code_context` first. **ONLY** if the results are insufficient should you call the read_file tool, which often causes context window to grow too quickly."

2. Expand surface area of tooling available to LLM: We currently only expose `request_code_context`, primary out of simplicity since it calls the same RAG function as our automatic `ContextPlan`. Instead, we could add other tools which used the RAG functions under the hood, but encouraged usage in different ways, e.g.
  - code_similarity_search: a tool which instructs the model to enter a rust snippet, which is then used by our dense vector similarity search to try finding code that might have semantic similarity.
  - symbol_lookup: the model enters a code item name and type (e.g. function, struct, etc), and we use a bm25 search to find the item.

### Poor feedback to model from tools

Some tools do not provide enough feedback to the model, sometimes causing it to
make multiple proposals or to attempt to have a proposal applied. Our current
workflow is for the model to make a proposal and, by default, the user must
review and explicitly approve the proposed code change before the patch is
applied to the files.

However, perhaps because the model does not get clear instructions or feedback
regarding these proposals, the model will sometimes try to submit a proposed
patch multiple times. 

### Repeated context searching of same code items.

Our current implementation of persisting context allows for context to be gathered while the model is in the loop of calling tools, and then after the next user message we remove most of the context that was includ3ed in the movel's previous tool calling loop. If the user is asking the model to perform some task that involves referenceing the same code items or files repeatedly, this leads to inefficient searching, in which the model tends to search for the same code items repeatedly.

It may be time to implement another phase of our long-term plan to handle context management, as described in `docs/active/plans/context-management/context-management.md` (and with other supporting documentation in the same directory). 

Additionally, we may want to consider some of the following solutions:

1. Persist a short reference (enough to use the find_item_exact tool like name, file, module path) and a short description of the contents. This could be an extra ssytem-driven step that happens in the background (perhaps with a sys-info message for now) betwen when the model ends its tool calling loop and the user responds. This would make it much easier for the model to navigate the code base in the future by providing a condensed reference/description so it could quickly and easily find the same code items again.

2. Implementing a way for the user to specify items to be kept as sticky context until removed again by the user. This is already part of our context plan, but it may be worth prioritizing so the user can have more directed influence over the items that are kept in context.
