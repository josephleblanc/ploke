# Evaluation / Protocol Record Audit

Scope: targeted audit of `crates/ploke-eval/src/protocol*`, `crates/ploke-eval/src/branch_evaluation*`, `crates/ploke-eval/src/runner.rs`, and direct eval/run artifact code (`record.rs`, `run_registry.rs`, `run_history.rs`, `operational_metrics.rs`, and Prototype 1 branch-evaluation persistence call sites).

## High-Level Findings

- `RunRegistration` is documented as the canonical identity/lifecycle/artifact-ref authority, but `RunRecord`, runner sidecars, batch summaries, execution logs, and `last-run.json` still repeat path, status, model, run-arm, and completion facts.
- `RunRecord` is the richest replay record, but setup, patch, packaging, timing, LLM, tool, and submission facts are also persisted as sidecar JSON/JSONL. This is useful for failure inspection, but authority is unclear after partial failures.
- Protocol records are double-wrapped: typed `ploke-protocol` output is persisted inside `StoredProtocolArtifact.output: Value`, while `StoredProtocolArtifact.artifact: Value` can hold generic step/procedure artifacts. Aggregates then normalize those values into another set of report rows.
- Branch evaluation persists derived metrics snapshots in the Prototype 1 evaluation report and a smaller summary in the branch registry. Those can become stale relative to the source `record.json.gz`.

## Priority Cleanup

1. Define explicit authorities: `RunRegistration` for discovery/lifecycle/artifact refs, `RunRecord` for replay archive, protocol artifacts for adjudication evidence, and Prototype 1 evaluation report for branch decision evidence.
2. Convert string status fields (`IndexingStatusArtifact.status`, execution steps, branch reason strings) into typed enums or coded records where they affect decisions.
3. Remove duplicate turn payloads by separating event journal from final turn summary, then embedding only one summary/reference in `RunRecord`.
4. Type the protocol artifact envelope by procedure instead of persisting three untyped `Value` blobs.
5. Add source fingerprints/digests to persisted derived records (`OperationalRunMetrics`, branch evaluation reports, protocol reports if saved) so stale projections are detectable.
