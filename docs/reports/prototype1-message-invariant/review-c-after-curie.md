# Prototype 1 Message Invariant Review C After Curie

## Verdict

Fail for absolute correctness. Curie improved the live happy path by adding `Sent`, `Packed`, `Received`, drop guards, and a `SelectedChild` carrier, and `cargo check -p ploke-eval --tests` passes. The model still permits sent-without-received states, same-runtime receipt, synthetic receipt from existing files, and materialization APIs that are not structurally gated by `Received<ChildPlan>`.

## Findings

1. Critical: `pack` still expresses sent before receipt, including filesystem-failure paths. `Open::pack` closes the sender and returns `(M::SenderClosed, Sent<M>)` as soon as the write closure succeeds ([inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:117), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:131)). For `ChildPlan`, that closed sender is `Parent<Planned>` ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:206)). In the live path, `_planned` is produced before `sent.read(...)` touches the filesystem ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:365), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:373)). If `read_child_plan_files` fails on any file read ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:447), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:457)), the error is mapped to the source and the typed `Failed<ChildPlan>` is discarded ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:373)). Minimal fix: do not expose `SenderClosed` until receipt succeeds, or make send/read/receive one transition that returns `Parent<Selectable>` plus `Received<ChildPlan>` only after successful receiver validation.

2. Critical: the same runtime can self-receive, and the current happy path does exactly that. `ChildPlan::ReceiverOpen` is `Parent<Ready>`, not a distinct recipient runtime state ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:210)). After sending, `receive_child_plan` reloads the same `ParentIdentity` from the same `repo_root` and receives the message in the same command invocation ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:377), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:415), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:426)). There is no runtime id, process boundary, or role distinction proving that the receiver is the intended other runtime. Minimal fix: model sender and receiver as distinct role/state carriers with runtime identity, and reject receive when sender runtime identity equals receiver runtime identity unless self-receipt is explicitly part of the protocol.

3. High: `Received<ChildPlan>` can be synthesized from existing scheduler/node files without a `Sent<ChildPlan>`. `Packed::<M>::read` is a public crate API that accepts an arbitrary `M::Buffer` and a read closure ([inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:257)). `receive_existing_child_plan` constructs `ChildPlanFiles` from the current parent identity and runnable nodes, reads those files, and receives it ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:386), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:394), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:395)). `resolve_next_child` takes this path whenever runnable child nodes already exist ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2567), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2578)). This proves file presence, not a prior message send. Minimal fix: make existing receipt load a durable sent-message record, not reconstruct the message from scheduler frontier.

4. High: materialization is not structurally gated by `Received<ChildPlan>` at the transition boundary. The live command path carries `Received<ChildPlan>` through `SelectedChild` and validates it before `C1::load` ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:302), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:320), [cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2910)). But `C1::load` itself is still `pub(crate)` and only requires `campaign_id`, `manifest_path`, `node_id`, and `repo_root` ([c1.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/c1.rs:378), [c1.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/c1.rs:380)). `MaterializeBranch` then consumes `C1` with no receipt capability ([cli_facing.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/cli_facing.rs:2924)). Minimal fix: require `Received<ChildPlan>` or a `SelectedChild`-like receipt carrier in the constructor that creates materializable `C1`.

5. Medium: recipient identity is partial, not exact. `ChildPlanFiles` records `parent_node_id` and `child_generation` ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:56)), and receive validation checks only `identity.node_id` plus `identity.generation + 1` ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:113), [parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:121)). `ParentIdentity` also contains `campaign_id`, `parent_id`, `branch_id`, `artifact_branch`, predecessor fields, and schema version ([identity.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/identity.rs:24)). Minimal fix: address the message to the full intended receiver identity, or to a dedicated receiver id, and validate all identity fields that define exact recipient semantics.

6. Medium: file locations are only partially structural. The file schemas use `At<F>` and `ChildPlanFiles` ties child node and runner-request paths to node ids ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:64), [parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:77), [parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:133)). But `ChildPlanFiles::for_parent` can be called with any manifest path, parent identity, and node slice inside the crate ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:65)), and `Packed::read` trusts that caller-provided buffer after existence reads. Minimal fix: make the durable message include the exact file-address set produced by the sender, with sender/receiver identities, instead of recomputing the address set from current scheduler state.

7. Medium: receive failure still loses the receiver state. `Packed::receive` passes `receiver` by value into `M::ready_receiver` ([inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:281), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:290)). On validation failure, `ReceiveError` preserves only `Failed<M>` and the source error ([inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:292), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:440)). Minimal fix: validate by reference before consuming the receiver, or return the receiver in `ReceiveError`.

## Positive Notes

`Sent<M>` and `Packed<M>` are no longer cloneable and now panic on unhandled drop ([inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:166), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:225), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:235), [inner.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/inner.rs:321)). `Parent<S>` also no longer derives `Clone` ([parent.rs](/home/brasides/code/ploke/crates/ploke-eval/src/cli/prototype1_state/parent.rs:42)). These are real improvements, but drop-time panics and call-site discipline do not prove the cross-runtime invariant absolutely.
